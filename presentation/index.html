Зачем разбивать приложение на пакеты

Плюсы

  1. Переиспользование кода
  2. Лучшее управление зависимостями
  3. Лучшее управление порядком загрузки кода
  4. Изоляция кода
  5. Для пакетов легко писать юнит-тесты

Минусы

  1. Надо создавать дескриптор пакета package.js
  2. Можно добавить новый файл и забыть прописать его в package.js и он не будет загружен

Пакетная система Meteor

  1. Изоморфная
  2. Может содержать плагины расширяющие build process
  3. Весь Meteor это набор packages

Создание нового пакета

  • meteor create --package username:packagename
  • Будет создан каталог и дескриптор пакета package.js
  • Windows: имя каталога не может содержать двоеточие

Структура package.js: Package.describe

        /* Information about this package */
        Package.describe({
          // Short two-sentence summary.
          summary: "What this does",
          // Version number.
          version: "1.0.0",
          // Optional.  Default is package directory name.
          name: "username:package-name",
          // Optional github URL to your source repository.
          git: "https://github.com/something/something.git",
        });

Публикация пакета

  • meteor publish

Ограничения

  • Версия обязательна
  • username должен совпадать с аккаунтом разработчика Meteor

Использование пакета локально

  • PACKAGE_DIRS
